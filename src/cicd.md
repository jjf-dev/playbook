# CI/CD Integration

This chapter describes how coding agents interact with the Asterinas CI pipeline and how to run agent-assisted workflows in automated contexts.

## How CI Treats Agent-Assisted PRs

PRs labeled `agent-assisted` run the same CI pipeline as all other PRs. There is no separate gate or bypass. The label exists for tracking and reviewer attention, not for relaxed checks.

All of the following must pass before merge:

| Job | What it checks |
|---|---|
| `build` | `cargo build --all-targets` on the full workspace |
| `test` | `cargo test --workspace` |
| `clippy` | `cargo clippy --all-targets -- -D warnings` |
| `fmt` | `cargo fmt --all -- --check` |
| `doc` | `cargo doc --no-deps --all` (no broken intra-doc links) |
| `ktest` | Kernel integration tests run inside QEMU |

If any job fails, the PR is not mergeable regardless of review approval.

## Running CI Locally Before Pushing

Always run the full check locally before pushing an agent-generated branch:

```sh
# One-shot pre-submit check (recommended)
make check

# Or run jobs individually:
cargo build --all-targets
cargo test --workspace
cargo clippy --all-targets -- -D warnings
cargo fmt --all -- --check
cargo doc --no-deps --all
```

Catching failures locally is faster than waiting for the CI queue.

## Using Agents Inside CI (Experimental)

> ⚠️ This feature is experimental. It is opt-in and not used on the main branch.

For certain well-defined tasks (doc generation, boilerplate expansion, changelog drafting), it is possible to invoke an agent inside a CI job. The pattern is:

1. A GitHub Actions workflow step calls a script that invokes the agent API.
2. The agent output is written to a file or applied as a patch.
3. Subsequent steps run the normal build and test jobs against the modified tree.
4. If the build passes, the workflow commits and pushes the changes to the branch.

### Example Workflow Step

```yaml
- name: Generate missing doc comments
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    python3 scripts/agent_doc_fill.py \
      --crate ostd \
      --output patch/doc_fill.patch

- name: Apply patch
  run: git apply patch/doc_fill.patch

- name: Verify build still passes
  run: cargo build --all-targets
```

### Safety Constraints for CI Agents

CI agent invocations must follow these constraints:

- **Read-only by default.** The agent may only write to designated output paths. It must not modify `Cargo.toml`, `Cargo.lock`, or any file outside the target crate.
- **No `unsafe` generation.** CI agent tasks are limited to tasks where unsafe code is not expected. The CI script should fail if `unsafe` appears in the agent output.
- **Deterministic scope.** The task given to the agent must be fully specified by the script—no interactive prompting, no ambient context from prior runs.
- **Human review before merge.** CI agent commits go to a branch, never directly to `main`. A human reviews the diff before merge.

## Labeling Conventions

| Label | Meaning |
|---|---|
| `agent-assisted` | Some or all of the code was generated by a coding agent |
| `agent-ci` | The change was produced by a CI-invoked agent workflow |
| `needs-safety-review` | Contains `unsafe` code; requires a designated safety reviewer |

Apply `agent-assisted` to any PR where an agent wrote a non-trivial portion of the code, even if you substantially edited the output afterward. This helps the team track agent quality over time.

## Tracking Agent Quality

The team periodically reviews agent-assisted PRs to assess:

- **Bug rate**: Did any agent-generated code introduce a bug that reached CI?
- **Review overhead**: Are agent PRs taking longer to review than human PRs?
- **Hallucination frequency**: How often do reviewers find nonexistent APIs in agent output?

Findings are fed back into this playbook. If you notice patterns—good or bad—open an issue or a PR against the book.
